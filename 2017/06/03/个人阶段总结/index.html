个人阶段总结A

公司技术团队不在大小，其重点是有无条例于准则。

接触过一些技术团队后，自己的一些感悟还是挺大的。

A团队

A公司，公司框架+node技术栈，svn管理代码，技术团队在8人左右，技术带头人是华为技术经理级别，所以对公司技术人员也是挺有要求和标准的，当初又一些地方不太理解他的做法，现在回头想来确实很有必要。总结如下：

  • 一.对于svn代码的提交有明确清晰的流程控制，提交说明必须简单清晰，多个任务必须分开提交，提交记录会以邮箱方式告知组长级别人员。
  • 二.每周有一次技术分享会，团队里每个人都必须参与。
  • 三.会对技术人员有考核，凡是不熟悉某一块的业务或者代码等，坚决不让其碰这一块代码
  • 四.对于团队日志，技术债，bug跟踪等等都在有道云笔记记录在案
  • 五.如果带头人对某一块的技术把控不了，坚决不用。
  • 六.提交代码前必须多方面测试过没问题。

B团队

B公司，公司项目用到了react这个框架，svn管理代码，技术团队6人左右，带头人不算懂技术，技术团队也是比较混乱，有外包驻场，有远程办公，也有公司本员工。项目也是经过几个外包公司的接管，代码比较混乱。对此，我的做法是在坚持自己好的习惯，严谨的做法外，慢慢地一点点规范起来。

  • 一.团队沟通不再用QQ，使用teambition，包括接口文档这些文件的管理
  • 二.使用svn管理代码，不允许u盘式的传递，提交也要写提交说明（至少我自己这么做）
  • 三.把一些以前外包公司不会注重的warning修复起来
  • 四.建立bug跟踪共享表格与技术债说明表格，以便查看历史问题
  • 有待努力更新中……
